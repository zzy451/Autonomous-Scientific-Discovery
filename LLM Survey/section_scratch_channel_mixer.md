## 2.1 FFN 基础结构与激活函数演进

**演进主线**: ReLU --> GELU --> Swish/SiLU --> SwiGLU --> GeGLU --> ReLU^2 --> PolyGLU

**涉及笔记**:
- Agarap_2018_ReLU.md — ReLU 基础
- Hendrycks_2016_GELU.md — GELU 激活函数
- Ramachandran_2017_Swish.md — Swish/SiLU 搜索发现
- Shazeer_2020_SwiGLU.md — SwiGLU 门控线性单元变体
- Zhang_2024_ReLU2Wins.md — ReLU^2 在稀疏 LLM 中的优势
- PolyGLU_2026.md — 状态条件激活路由

### 分支定义与存在理由

激活函数是 FFN 中唯一的非线性来源，决定了 token 级特征变换的表达能力边界。从 ReLU 的分段线性到 GELU 的平滑概率门控，再到 SwiGLU 的乘性门控交互，激活函数的每一次演进都直接影响模型的训练动态、收敛速度和最终性能。这条演进线是理解 Channel Mixer 设计哲学的起点：如何在计算开销可控的前提下最大化逐 token 的非线性变换能力。

### 技术介绍

- **ReLU**: $\text{ReLU}(x) = \max(0, x)$。最简单的激活函数，计算高效但存在"神经元死亡"问题（负区间梯度为零）。在早期 Transformer 中广泛使用。
- **GELU** (Hendrycks & Gimpel, 2016): $\text{GELU}(x) = x \cdot \Phi(x)$，其中 $\Phi$ 为标准正态 CDF。将激活解释为随机正则化的期望，提供平滑非线性。GPT-2/BERT 的标准选择。
- **Swish/SiLU** (Ramachandran et al., 2017): $\text{Swish}(x) = x \cdot \sigma(\beta x)$。通过自动搜索发现，非单调且处处可导。$\beta=1$ 时即为 SiLU。
- **SwiGLU** (Shazeer, 2020): $\text{SwiGLU}(x, W, V, b, c) = \text{Swish}(xW + b) \otimes (xV + c)$。结合 Swish 激活与 GLU 门控的三矩阵结构，在同参数量下显著优于 ReLU/GELU FFN。LLaMA、PaLM 等模型的标准选择。
- **GeGLU**: 将 SwiGLU 中的 Swish 替换为 GELU，$\text{GeGLU}(x) = \text{GELU}(xW) \otimes (xV)$。Gemma 等模型采用。
- **ReLU^2** (Zhang et al., 2024): $\text{ReLU}^2(x) = (\max(0,x))^2$。关键发现：ReLU^2 自然产生极高稀疏度（90%+激活为零），使得推理时可跳过大量计算，在稀疏 LLM 场景下性能与 SwiGLU 相当但推理更快。
- **PolyGLU** (2026): 状态条件激活路由，激活函数不再是静态的，而是根据输入状态动态选择激活模式。打破了"一个激活函数服务所有 token"的范式。

### 理论支撑

- **万能近似定理** (Universal Approximation Theorem): 单隐层网络配合非线性激活即可近似任意连续函数。激活函数的选择影响近似效率而非近似能力本身。
- **稀疏性与信息瓶颈**: ReLU 系激活的零激活区域天然实现特征选择，与信息瓶颈理论中压缩无关信息的目标一致。ReLU^2 将这一性质推向极端。
- **平滑优化景观**: GELU/Swish 的平滑性避免了 ReLU 在零点的梯度不连续，有助于优化器找到更好的局部最优。

### 待解决问题

- **SwiGLU 的理论解释缺失**: SwiGLU 在实验中一致优于 GELU FFN，但缺乏严格的理论解释。为什么乘性门控（而非加性）对 Transformer FFN 特别有效？是因为梯度流改善、特征选择能力还是隐式正则化？
- **激活函数与模型规模的交互**: 不同激活函数在不同模型规模下是否有不同的最优选择？目前的比较主要集中在中等规模。
- **稀疏性-精度权衡的理论边界**: ReLU^2 的高稀疏性何时会导致信息损失？是否存在任务依赖的最优稀疏度？

### 对未来的影响

- **PolyGLU 等状态条件激活可能打破静态激活范式**: 如果输入条件化激活被验证有效，未来 FFN 设计将从"选择哪个激活函数"转向"如何设计激活选择机制"。
- **稀疏激活与硬件协同设计**: ReLU^2 的高稀疏性暗示未来硬件（如稀疏加速器）与激活函数的联合优化方向。
- **激活函数搜索的自动化**: Swish 由 NAS 发现的先例表明，更大规模的激活函数自动搜索（结合 LLM 场景）可能发现更优解。

### 潜在技术方向

- **自适应激活函数**: 可学习参数化激活，如 PReLU 的推广，让模型自主学习每层甚至每神经元的最优激活形式。
- **输入条件化激活**: 激活函数的行为依赖于输入 token 的语义特征（PolyGLU 的思路推广）。
- **激活-注意力联合设计**: 将 Attention 的动态选择思想引入 FFN 激活，实现跨 token 条件化的特征变换。

## 2.2 GLU 门控机制

**演进主线**: GLU --> Bilinear --> ReGLU --> SwiGLU --> GeGLU，三矩阵结构

**涉及笔记**:
- Dauphin_2017_GLU.md — 原始 GLU 提出
- Shazeer_2020_SwiGLU.md — GLU 变体的系统比较
- Ramachandran_2017_Swish.md — Swish 激活（SwiGLU 的组成部分）

### 分支定义与存在理由

GLU 门控机制是从标准 FFN（两矩阵结构 $W_1, W_2$）到现代 FFN（三矩阵结构 $W_{gate}, W_{up}, W_{down}$）的关键转折。门控引入乘性交互（element-wise product），使得 FFN 具备了类似 LSTM 门控的信息过滤能力：一条路径生成候选特征，另一条路径决定哪些特征被保留。这种机制从根本上改变了 FFN 的信息处理方式，从简单的"激活-投影"变为"生成-选择-投影"。

### 技术介绍

- **标准 FFN**: $\text{FFN}(x) = W_2 \cdot \sigma(W_1 x + b_1) + b_2$，两矩阵结构。
- **GLU** (Dauphin et al., 2017): $\text{GLU}(x) = (xW + b) \otimes \sigma(xV + c)$。原始提出用于卷积语言模型，$\sigma$ 为 sigmoid。右半部分作为门控信号，控制左半部分的信息流。
- **Bilinear**: $\text{Bilinear}(x) = (xW) \otimes (xV)$，去掉激活函数的纯乘性交互。
- **ReGLU**: $\text{ReGLU}(x) = \text{ReLU}(xW) \otimes (xV)$，用 ReLU 替代 sigmoid。
- **SwiGLU**: $\text{SwiGLU}(x) = \text{Swish}(xW) \otimes (xV)$，Shazeer 2020 的系统实验表明 SwiGLU 在多数设置下最优。
- **GeGLU**: $\text{GeGLU}(x) = \text{GELU}(xW) \otimes (xV)$。
- **三矩阵结构的参数量**: GLU 变体引入第三个矩阵使参数从 $2dh$ 增加到 $3dh$，实践中通过将 $h$ 缩小为 $\frac{2}{3} \times 4d$（而非标准 $4d$）来保持总参数量不变。

### 理论支撑

- **信息门控理论**: 门控机制可视为对隐层表示的软注意力（soft attention），与 LSTM 中遗忘门/输入门的信息过滤思想同源。乘性交互允许网络学习特征之间的高阶关系。
- **LSTM 门控类比**: LSTM 的成功表明门控是序列建模中控制信息流的有效范式。GLU 将这一思想从循环结构迁移到前馈结构。
- **梯度流分析**: GLU 的乘性结构提供了更直接的梯度路径（类似残差连接的效果），可能缓解深层网络的梯度消失问题。

### 待解决问题

- **为何三矩阵 GLU 优于等参数量的宽 FFN？** 在总参数量相同的情况下，$3 \times \frac{2}{3}h$ 的 GLU 结构为何优于 $2 \times h$ 的标准 FFN？是因为乘性交互的表达能力还是优化景观的改善？
- **门控信号的可解释性**: 门控值的分布和语义含义尚不清楚。门控是否在学习有意义的特征选择？
- **最优门控激活的选择**: SwiGLU vs GeGLU 的差异微小，是否存在更优的门控激活？

### 对未来的影响

- **三矩阵结构已成为事实标准**: LLaMA、Mistral、Qwen、DeepSeek 等主流模型均采用 SwiGLU/GeGLU，两矩阵 FFN 在前沿模型中已基本被淘汰。
- **门控思想向其他模块扩散**: 门控机制可能启发 Attention 中的类似设计（如 Gated Attention）。
- **与 MoE 的结合**: 门控 FFN + MoE 路由 = 双层门控，这种双层选择机制的交互效应值得深入研究。

### 潜在技术方向

- **多级门控**: 在 FFN 内部引入多层级的门控（粗粒度通道选择 + 细粒度激活门控）。
- **跨层门控共享**: 不同层的门控信号是否可以共享或条件化，减少冗余。
- **动态门控维度**: 门控的粒度从 element-wise 扩展到 group-wise 或 channel-wise，在表达能力和计算效率之间寻找新的平衡点。

## 2.3 MoE 基础理论与路由策略

**演进主线**: Top-k --> Expert Choice --> Aux-Loss-Free --> Soft MoE --> DirMoE --> ReMoE --> Sigma-MoE

**涉及笔记**:

- Lepikhin_2021_GShard.md — GShard 分布式 MoE
- Fedus_2022_SwitchTransformer.md — Switch Transformer (Top-1)
- Zoph_2022_STMoE.md — ST-MoE 稳定训练
- Zhou_2022_ExpertChoice.md — Expert Choice 路由
- Wang_2024_AuxLossFree.md — 无辅助损失负载均衡
- Puigcerver_2024_SoftMoE.md / Puigcerver_2024_SoftMoE_detailed.md — Soft MoE
- Wang_2024_ReMoE.md — ReLU 路由 MoE
- DirMoE_2025.md — Dirichlet 路由
- Hu_2024_SigmaMoE.md — Sigma-MoE
- Zhong_2024_Lory.md — 全可微 MoE
- DenseBP_2025_SparseMoE.md — Dense Backpropagation

### 分支定义与存在理由

路由策略是 MoE 的灵魂组件——它决定每个 token 被发送到哪些专家，直接影响负载均衡、专家专业化程度和训练稳定性。从 Shazeer 2017 的 Top-k + 辅助损失，到 Expert Choice 翻转路由方向，再到 Aux-Loss-Free 消除辅助损失的干扰，路由策略的演进反映了 MoE 领域对"如何分配计算"这一核心问题的持续探索。条件计算是扩展模型容量同时控制推理成本的核心手段。

### 技术介绍

- **Top-k 路由** (Shazeer, 2017): 对每个 token，通过线性层计算所有专家的得分，选取得分最高的 $k$ 个专家。$g(x) = \text{TopK}(\text{softmax}(Wx + \epsilon))$，其中 $\epsilon$ 为噪声（训练时）。Top-2 是原始设定。
- **Top-1 (Switch)** (Fedus, 2022): 极端简化为每 token 仅选 1 个专家。降低计算成本但增加负载不均衡风险。需要更强的负载均衡辅助损失。
- **Expert Choice** (Zhou, 2022): 翻转路由方向——不是 token 选专家，而是每个专家从所有 token 中选取固定数量的 token。天然保证完美负载均衡，但导致不同 token 被处理的专家数量不同（有些 token 可能不被任何专家选中）。
- **Aux-Loss-Free** (Wang et al., 2024/DeepSeek-V3): 通过在路由层引入可学习偏置项（bias term），在不使用辅助损失的情况下实现负载均衡。偏置项在训练中动态调整，将 token 从过载专家引导到空闲专家。避免了辅助损失对主任务梯度的干扰。
- **Soft MoE** (Puigcerver, 2024): 完全消除离散路由。每个 token 被软分配到所有专家（加权组合），每个专家处理所有 token 的加权混合。全可微但丧失了稀疏计算优势。
- **DirMoE** (2025): 使用 Dirichlet 分布对路由概率建模，提供更灵活的多专家分配模式。
- **ReMoE** (Wang et al., 2024): 用 ReLU 替代 softmax 进行路由，天然产生稀疏路由（大量专家权重为零），无需显式 Top-k 选择。
- **Sigma-MoE** (Hu, 2024): 使用 sigmoid 路由替代 softmax，允许专家权重独立计算（非竞争性），改善专家利用率。
- **Lory** (Zhong, 2024): 全可微 MoE，通过合并专家参数实现连续路由。
- **Dense Backpropagation** (2025): 在前向传播中保持稀疏，但在反向传播中使用稠密梯度，改善专家训练信号。

### 理论支撑

- **Bengio 条件计算理论** (2013/2015): 首次系统论述条件计算——对不同输入使用不同计算路径，以在模型容量和计算成本之间解耦。MoE 是条件计算的具体实现。
- **混合模型理论**: MoE 可视为混合高斯模型的深度学习推广，路由器对应混合权重，专家对应混合分量。EM 算法的思想在训练中隐式体现。
- **负载均衡与社会福利**: 路由问题可类比为经济学中的资源分配问题——如何在不损害整体效用的前提下实现公平分配。

### 待解决问题

- **路由坍缩** (Routing Collapse): 训练中少数专家获得大部分 token，其余专家退化。虽有多种缓解策略，但根本原因（是优化景观问题还是路由器容量不足？）尚不明确。
- **辅助损失的最优权重**: 辅助损失过大损害主任务性能，过小无法保证均衡。目前主要依赖经验调参。
- **离散 vs 连续路由的根本权衡**: Soft MoE 全可微但失去稀疏性；Top-k 稀疏但不可微。是否存在兼顾两者的路由方案？
- **路由决策的语义一致性**: 相似语义的 token 是否应被路由到相同专家？目前缺乏对路由决策语义含义的系统研究。

### 对未来的影响

- **Aux-Loss-Free 路由可能成为新标准**: DeepSeek-V3 的成功验证表明，消除辅助损失既简化了超参调优又改善了性能，这一方向可能取代传统辅助损失方法。
- **路由与 token 重要性的结合**: 未来路由器可能不仅决定"哪个专家"，还决定"是否需要计算"（与 MoD 思想融合）。
- **多模态路由**: 在多模态模型中，不同模态的 token 可能需要不同的路由策略。

### 潜在技术方向

- **层次化路由**: 先粗粒度分组再细粒度选择，减少路由器的决策空间。
- **路由器预训练**: 用无监督方法预训练路由器，给予更好的初始化。
- **自适应 Top-k**: 不同 token 使用不同的 $k$ 值，简单 token 用少量专家，复杂 token 用更多专家。
- **路由信号的跨层传播**: 浅层路由决策信息传递给深层，实现跨层协调的专家分配。

## 2.4 MoE 专家设计

**核心议题**: 粗粒度 vs 细粒度、Shared Expert vs No Shared、同构 vs 异构

**涉及笔记**:
- Dai_2024_DeepSeekMoE.md — 细粒度专家 + Shared Expert

- DeepSeekAI_2024_DeepSeekV3.md — DeepSeek-V3 (Aux-Loss-Free + Shared Expert)
- Databricks_2024_DBRX.md — DBRX 细粒度专家 (16 选 4)
- Jiang_2024_Mixtral.md — Mixtral 粗粒度 (8 选 2)
- He_2024_MillionExperts.md — 百万级专家
- SharedExpert_vs_NoShared_2025.md — Shared Expert 设计分析
- Qwen3_2025_MoE.md / Qwen_2025_Qwen3.md / Qwen3_no_shared_expert.md — Qwen3 无 Shared Expert
- ERNIE45_heterogeneous_MoE.md — ERNIE 4.5 异构专家
- Meta_2025_Llama4.md — Llama 4 MoE

### 分支定义与存在理由

专家的粒度（单个专家的大小和总数量）和共享策略（是否设置 Shared Expert）直接决定了 MoE 模型的容量-效率权衡。这是 MoE 架构设计中最核心的结构性决策之一——它影响专家专业化程度、参数利用率、通信开销和推理效率。2024-2025 年间这一方向出现了显著分歧（DeepSeek 的 Shared Expert vs Qwen3 去掉 Shared Expert），反映了社区对最优设计的认知仍在演变。

### 技术介绍

- **粗粒度专家** (8-16 个专家): Mixtral (8 选 2)、Grok-1 (8 选 2)。每个专家是完整的 FFN（维度与 dense 模型相同）。优势：通信开销小，实现简单。劣势：专家专业化受限，粒度太粗无法精细分工。
- **细粒度专家** (64-256 个专家): DeepSeekMoE (64 选 6)、DBRX (16 选 4)、DeepSeek-V2 (160 选 6)。每个专家是 FFN 的子集（缩小的隐层维度）。优势：更灵活的专家组合，更高的专业化。劣势：路由决策空间更大，通信更复杂。
- **百万级专家** (He et al., 2024): 将专家粒度推到极端，每个专家极小（接近单个神经元组）。与乘积密钥（product key）路由结合，实现高效检索。
- **Shared Expert**: DeepSeekMoE/V2/V3 设置 1-2 个始终被激活的共享专家，处理所有 token 共有的通用特征。路由专家专注于特定任务或领域。类比：共享专家 = 通用基底，路由专家 = 专业化增量。
- **No Shared Expert**: Qwen3 (2025) 明确去掉 Shared Expert，论证在全局批次均衡策略下 Shared Expert 不再必要。路由专家本身可以学会通用特征。
- **异构专家** (ERNIE 4.5): 不同专家具有不同的架构（不同隐层维度、不同层数），而非同构的缩放版本。允许模型为不同类型的输入分配不同复杂度的计算。

### 理论支撑

- **分而治之原理**: 细粒度专家实现了更精细的"分治"——将特征空间划分为更多子区域，每个专家专精一个子区域。
- **共享-特有分解**: Shared Expert 对应信号处理中的共模分量提取，路由专家对应差模分量。这种分解在数学上类似低秩+稀疏分解 (robust PCA)。
- **专家组合的组合爆炸优势**: $C(N, k)$ 的组合数随 $N$ 增大而指数增长，细粒度专家通过更多组合覆盖更大的功能空间。

### 待解决问题

- **Shared Expert 到底有没有必要？** DeepSeek 和 Qwen3 的分歧表明社区尚无共识。是否取决于路由策略（序列级均衡 vs 全局批次均衡）？
- **最优专家粒度**: 是否存在与模型规模相关的最优专家数量/大小比？从 8 到 256 到百万，在什么点收益递减？
- **异构专家的路由**: 当专家具有不同容量时，路由器如何学会为复杂 token 分配"大"专家、为简单 token 分配"小"专家？
- **专家冗余**: 大量专家中是否存在功能高度重叠的冗余专家？如何度量和消除？

### 对未来的影响

- **专家粒度将继续向细粒度方向演进**: DBRX 和 DeepSeek 的成功表明细粒度是更优方向，但需要更好的路由和通信方案支撑。
- **Shared Expert 的命运取决于路由技术进步**: 如果路由器足够智能（能让部分路由专家自然学会通用特征），Shared Expert 可能变得不必要。
- **异构 MoE 可能成为多模态模型的标准**: 不同模态（文本、图像、代码）天然需要不同复杂度的专家处理。

### 潜在技术方向

- **自适应专家生长/剪枝**: 训练过程中动态调整专家数量——利用率低的专家被合并，高负载专家被分裂。
- **专家架构搜索 (Expert NAS)**: 自动搜索每个专家的最优架构（层数、宽度、激活函数）。
- **专家参数共享**: 不同专家共享部分参数（如 $W_{down}$），仅在特化部分不同，减少总参数量。
- **跨层专家复用**: 不同层的专家是否可以共享，减少参数冗余（类似 Universal Transformer 思想）。

## 2.5 MoE 工程实践

**核心议题**: 负载均衡策略、Expert Parallelism、通信优化、FP8 训练

**涉及笔记**:

- DeepSeekAI_2024_DeepSeekV3.md — FP8 训练 + Aux-Loss-Free
- MoE_DoublePenalty_2026.md — MoE 推理双重惩罚
- Wei_2024_SkyworkMoE.md — Skywork-MoE 训练技巧
- Muennighoff_2024_OLMoE.md — 开源 MoE 实践
- Tencent_2024_HunyuanLarge.md — Hunyuan-Large 工程

- xAI_2024_Grok1.md — Grok-1 (314B MoE)
- Lieber_2024_Jamba.md — Jamba 混合架构

### 分支定义与存在理由

MoE 的理论优势（大容量、低计算成本）需要精密的工程手段才能在大规模集群上兑现。分布式 MoE 面临独特的挑战：All-to-All 通信瓶颈、专家负载不均导致的 GPU 空闲、混合精度训练中的数值不稳定、以及推理时的"双重惩罚"（内存带宽 + 通信延迟）。这一分支记录将理论转化为可用系统的工程实践。

### 技术介绍

- **Expert Parallelism (EP)**: 将不同专家分布在不同 GPU 上，每个 GPU 只持有部分专家。需要 All-to-All 通信——训练时每个 token 被发送到其被路由的专家所在 GPU，处理后再发送回来。GShard 首次在 Transformer 中实现大规模 EP。
- **DeepEP** (DeepSeek, 2025): 专为 MoE 设计的 Expert Parallelism 通信库。核心优化：(1) 将 All-to-All 分解为 Dispatch + Combine 两个阶段，与计算重叠；(2) 利用 NVLink/NVSwitch 的拓扑感知路由减少跨节点通信；(3) 支持低精度通信（FP8）减少带宽需求。
- **负载均衡工程**: (1) Aux Loss: 添加辅助损失惩罚不均匀分配，但干扰主梯度；(2) Random Routing: 对剩余容量的 token 随机分配；(3) Capacity Factor: 设置每个专家的最大 token 数，溢出 token 被丢弃或重路由；(4) Aux-Loss-Free Bias: DeepSeek-V3 的偏置项方法。
- **FP8 训练** (DeepSeek-V3): 首次在 MoE 大模型中成功应用 FP8 混合精度训练。关键：(1) 仅在 FFN 专家的 GEMM 中使用 FP8，Attention 和路由器保持 BF16；(2) Per-group scaling 而非 per-tensor scaling 保持精度；(3) 节省约 40% 通信带宽。
- **MoE 推理的双重惩罚** (2026): (1) 内存惩罚：全部专家参数需加载到内存，但每次推理仅使用其中一小部分，参数利用率极低；(2) 通信惩罚：分布式推理中 All-to-All 延迟与专家数量和集群规模成正比。这导致 MoE 的训练效率优势不等于推理效率优势。
- **训练稳定性** (ST-MoE): 路由器的 softmax 输出在训练中容易产生数值不稳定（梯度爆炸/消失）。ST-MoE 的解决方案：Router z-loss（惩罚路由器 logits 的绝对值过大）。

### 理论支撑

- **通信复杂度分析**: All-to-All 通信量为 $O(N \cdot H / P)$（$N$ 为 token 数，$H$ 为隐层维度，$P$ 为 GPU 数），在大规模集群中通信成本可能超过计算成本。
- **Amdahl 定律**: MoE 的加速比受限于非并行部分（路由器计算、All-to-All 通信），这决定了 Expert Parallelism 的可扩展性上限。
- **数值精度与模型质量的权衡**: FP8 的动态范围有限（E4M3 仅 240），需要仔细的缩放策略避免溢出和精度损失。

### 待解决问题

- **推理时 MoE 是否真的省计算？** 虽然每 token 的 FLOP 更少，但内存带宽利用率极低（大量参数被加载但未使用）。在内存瓶颈的推理场景下，MoE 可能反而更慢。
- **超大规模 MoE 的通信可扩展性**: 当专家数量达到数百甚至数千时，All-to-All 通信如何随集群规模扩展？
- **异构硬件上的 MoE**: 不同 GPU 之间性能差异如何影响 EP 的效率？
- **MoE 的故障容错**: 单个 GPU 故障导致部分专家不可用时，如何优雅降级？

### 对未来的影响

- **推理优化将成为 MoE 的核心研究方向**: 训练侧的工程问题已基本解决（DeepEP, FP8），但推理侧的双重惩罚仍是根本性挑战。
- **MoE 与推理硬件的协同设计**: 未来加速器可能专门为 MoE 的稀疏访存模式和 All-to-All 通信优化。
- **小型 MoE 的兴起**: OLMoE 等工作表明 MoE 在中小规模也可行，可能改变"MoE 只适合超大模型"的认知。

### 潜在技术方向

- **Expert Offloading**: 将不活跃专家卸载到 CPU/SSD，仅在需要时加载到 GPU，解决内存瓶颈。
- **Expert Caching**: 预测未来 token 可能需要的专家并提前加载，隐藏 I/O 延迟。
- **通信-计算联合调度**: 在 All-to-All 通信进行时同时计算 Shared Expert，最大化硬件利用率。
- **MoE 特化芯片**: 设计支持高效稀疏矩阵运算和动态路由的专用硬件。

## 2.6 条件计算理论

**核心概念**: Conditional Computation、Mixture-of-Depths、Early Exit

**涉及笔记**:
- Bengio_2013_ConditionalComputation.md — 条件计算原始理论
- Raposo_2024_MoD.md — Mixture-of-Depths
- DuoLLM_2024.md — Duo-LLM 自适应计算

### 分支定义与存在理由

条件计算是 MoE 的上游理论基础，它提出了一个更一般的问题：对于给定输入，模型是否需要执行所有计算？MoE 回答的是"哪些专家"，而条件计算更广泛地追问"是否计算"和"计算多少"。Mixture-of-Depths 将这一思想从"专家选择"扩展到"层选择"——某些 token 可以跳过整个 Transformer 层。这一分支代表了从"固定计算图"到"自适应计算图"的范式转变。

### 技术介绍

- **条件计算** (Bengio, 2013/2015): 核心思想：大型神经网络中，对于任何给定输入，只有一小部分参数/计算是"相关"的。如果能够学会为每个输入动态选择计算子图，就能在不增加推理成本的情况下大幅增加模型容量。Bengio 将此框架化为强化学习问题——用策略网络决定激活哪些计算单元。
- **Mixture-of-Depths (MoD)** (Raposo et al., 2024): 为每个 Transformer 层训练一个轻量级路由器，决定每个 token 是否经过该层的完整计算（Attention + FFN）还是直接通过残差连接跳过。关键设计：设定每层的"容量"（如仅处理 50% 的 token），路由器选择最需要计算的 token。实验表明可在保持性能的前提下减少 50%+ 的 FLOP。
- **Duo-LLM** (2024): 为每个 token 维护两条计算路径（轻量级 + 完整计算），根据 token 的"难度"动态选择。框架化了自适应计算研究，提供了系统化的实验方法论。

### 理论支撑

- **Bengio 条件计算理论框架**: (1) 模型容量与计算成本的解耦是可行的；(2) 稀疏激活是实现解耦的核心手段；(3) 路由决策本身应该是轻量级的（否则路由开销抵消计算节省）。
- **自适应计算时间** (Adaptive Computation Time, Graves 2016): Universal Transformer 中的 ACT 机制——模型自主决定每个 token 需要多少步计算。MoD 可视为 ACT 的离散化版本。
- **计算资源的不均匀分配假说**: 自然语言中，不同 token 的"信息密度"差异巨大——功能词（the, is）可能只需少量计算，而关键语义词（否定词、专有名词）需要更多计算。条件计算利用了这一不均匀性。

### 待解决问题

- **路由器能否准确估计 token 难度？** 在深层网络中，token 的"重要性"依赖于全局上下文，但路由器只能看到局部信息。这种信息不对称是否限制了条件计算的有效性？
- **跳过计算的累积效应**: 如果一个 token 在多层被跳过，其表示是否会逐渐偏离正常演化轨迹？跨层跳过的协调问题尚未解决。
- **训练与推理的一致性**: 训练时使用可微松弛（如 Gumbel-Softmax）近似离散路由决策，推理时使用硬路由。这种不一致性是否导致性能损失？
- **与 KV Cache 的交互**: 被跳过层的 token 不生成新的 KV，这对后续层的 Attention 计算有何影响？

### 对未来的影响

- **MoE + MoD 的融合**: 同时在专家维度（哪些专家）和深度维度（哪些层）进行条件计算，实现双重稀疏。DeepSeek 等模型已在探索这一方向。
- **自适应推理预算**: 根据输入的复杂度动态调整总计算量——简单查询用少量计算快速回答，复杂推理任务用更多计算。这与 Test-time Compute Scaling 的思想高度一致。
- **条件计算作为通用效率框架**: 不仅限于 FFN/层级跳过，还可以扩展到 Attention 头的选择性激活、嵌入维度的动态调整等。

### 潜在技术方向

- **联合深度-宽度条件计算**: 同时决定"跳过哪些层"和"在保留的层中使用哪些专家/通道"。
- **全局预算分配器**: 一个前置的轻量级网络，在看到完整输入后为每层/每专家分配计算预算。
- **条件计算的课程学习**: 训练初期使用完整计算保证充分学习，逐步引入稀疏化让模型学会"哪些计算可以省略"。
- **与推测解码 (Speculative Decoding) 的结合**: 用条件计算框架实现更高效的推测验证。

## 2.7 FFN 稀疏性与压缩

**核心议题**: 上下文稀疏性、MoEfication、专家剪枝、ReLU^2 稀疏性

**涉及笔记**:
- Liu_2023_DejaVu.md — Deja Vu 上下文稀疏性
- Zhang_2022_MoEfication.md — FFN 到 MoE 的转换
- ExpertPruning_2024_Survey.md — 专家级剪枝综述
- Zhang_2024_ReLU2Wins.md — ReLU^2 稀疏性
- MoE_DoublePenalty_2026.md — MoE 推理效率分析

### 分支定义与存在理由

推理效率是 LLM 部署的关键瓶颈，而 FFN 层通常占据模型参数的 2/3 和计算量的大部分。FFN 稀疏性研究揭示了一个关键现象：对于给定输入，FFN 中大量神经元的激活值接近零——这意味着大部分计算是"浪费的"。利用这种稀疏性（无论是通过激活函数设计、后训练转换还是显式剪枝），可以在不显著损失精度的前提下大幅减少推理计算量。这是推理效率优化的关键方向。

### 技术介绍

- **上下文稀疏性 (Contextual Sparsity)** — Deja Vu (Liu et al., 2023): 核心发现：对于特定输入 token，LLM 中高达 95% 的参数可以在不影响输出的情况下被跳过。关键区别于静态稀疏性——哪些参数可以跳过依赖于输入（"上下文相关"）。方法：训练一个轻量级预测器（MLP head），根据前一层的激活预测当前层哪些神经元会被激活，仅计算预测为活跃的神经元。实现了近 2 倍加速。
- **MoEfication** (Zhang et al., 2022): 将训练好的 Dense FFN 后训练转换为 MoE 结构。方法：(1) 将 FFN 的神经元聚类为多个"专家"（基于激活模式的共现性）；(2) 训练一个轻量级路由器选择相关专家。无需从头训练 MoE，可直接复用已有的 Dense 模型权重。
- **专家剪枝** (Expert Pruning): 在已训练的 MoE 模型中，移除利用率低或贡献小的专家。方法：(1) 基于路由频率的剪枝（移除很少被选中的专家）；(2) 基于性能影响的剪枝（移除对输出贡献最小的专家）；(3) 基于相似度的专家合并（功能重叠的专家合并为一个）。
- **ReLU^2 稀疏性**: ReLU^2 激活在 LLM 中产生 90%+ 的零激活率。与 GELU/SiLU 的"软"稀疏不同（激活值很小但非零），ReLU^2 产生"硬"稀疏（精确为零）。硬稀疏可以被硬件直接利用——跳过零值的乘法运算。

### 理论支撑

- **彩票假设 (Lottery Ticket Hypothesis)**: 大型网络中存在小型子网络（"中奖彩票"），其性能可以匹配完整网络。FFN 稀疏性可视为彩票假设在激活层面的体现。
- **稀疏表示理论**: 自然信号在适当基下通常具有稀疏表示（压缩感知理论）。LLM 的隐层表示继承了这一性质。
- **激活函数与稀疏性的关系**: ReLU 系激活天然产生稀疏性（负值归零），而 GELU/SiLU 的平滑性虽然有利于训练，但牺牲了推理时可利用的稀疏性。ReLU^2 通过平方运算放大了这种稀疏性。

### 待解决问题

- **稀疏预测器的准确性**: Deja Vu 的稀疏预测器需要在速度和准确性之间权衡。错误的稀疏预测会导致质量损失，过保守的预测则无法充分加速。
- **训练时稀疏 vs 推理时稀疏**: 目前大多数方法是后训练稀疏化。是否可以在训练时直接引入稀疏性约束，让模型"学会稀疏"？
- **跨层稀疏模式的相关性**: 不同层的稀疏模式是否相关？如果是，可以用浅层的稀疏模式预测深层，进一步减少预测开销。
- **稀疏性与量化的交互**: 稀疏化和量化都是推理优化手段，两者的联合优化是否存在协同效应？

### 对未来的影响

- **"先 Dense 训练，后稀疏推理"可能成为标准范式**: 用 SwiGLU 训练以获得最优性能，推理时转换为 ReLU^2 或 MoEfication 以获得效率。
- **硬件-软件协同的稀疏加速**: 稀疏计算的效率高度依赖硬件支持（稀疏矩阵乘法、条件执行）。未来 GPU/TPU 可能原生支持 FFN 稀疏性。
- **MoEfication 模糊了 Dense 和 MoE 的边界**: 如果任何 Dense 模型都可以高效转换为 MoE，那么"Dense vs MoE"可能不再是架构设计时的二元选择。

### 潜在技术方向

- **渐进式稀疏化**: 训练中逐步增加稀疏度目标，从完全稠密到高度稀疏。
- **任务自适应稀疏度**: 不同任务/领域使用不同的稀疏度——翻译任务可能需要更多激活神经元，而简单分类可以更稀疏。
- **稀疏性的可微学习**: 用可微的稀疏度控制机制（如 $\ell_1$ 正则化的软阈值化）替代硬 Top-k 选择。
- **稀疏 FFN 与 Attention 的联合稀疏化**: 同时稀疏化 FFN 和 Attention（如 sparse attention + sparse FFN），实现全链路稀疏推理。

## 2.8 FFN 替代方案

**核心探索**: KAN (Kolmogorov-Arnold Networks)、Duo-LLM

**涉及笔记**:
- Liu_2024_KAN.md — Kolmogorov-Arnold Networks
- DuoLLM_2024.md — Duo-LLM 自适应计算框架

### 分支定义与存在理由

标准 MLP（线性变换 + 非线性激活）自 1960 年代以来一直是神经网络的默认特征变换模块。虽然 GLU 门控和 MoE 在 MLP 框架内做了大量创新，但 MLP 的基本范式——固定激活函数 + 可学习线性权重——从未被根本性质疑。KAN 和 Duo-LLM 代表了探索 MLP 之外的特征变换范式的尝试，试图回答一个根本问题：MLP 是 Transformer 中唯一合理的 Channel Mixer 吗？

### 技术介绍

- **KAN (Kolmogorov-Arnold Networks)** (Liu et al., 2024): 基于 Kolmogorov-Arnold 表示定理——任何多元连续函数都可以表示为一元函数的有限组合。KAN 的核心创新：将可学习性放在激活函数上（用 B-spline 参数化），而非权重矩阵上。
  - 结构：$\text{KAN}(x) = \sum_q \phi_q(\sum_p \psi_{p,q}(x_p))$，其中 $\phi_q$ 和 $\psi_{p,q}$ 都是可学习的一元函数（B-spline）。
  - 与 MLP 的本质区别：MLP 的激活函数是固定的（如 ReLU/GELU），可学习部分在线性权重上；KAN 的权重固定为组合结构，可学习部分在激活函数上。
  - 优势：在低维科学计算任务上，KAN 展现出更好的近似效率和可解释性。
  - 在 LLM 中的挑战：(1) B-spline 计算成本远高于矩阵乘法；(2) 目前缺乏高效 GPU 实现；(3) 在高维大规模设置下是否仍保持优势尚不明确。

- **Duo-LLM** (2024): 不是替代 MLP，而是构建"双 MLP"框架——每层维护轻量级和标准两套 FFN，根据 token 复杂度动态选择。
  - 轻量级 FFN：缩小隐层维度的 MLP，处理"简单"token。
  - 标准 FFN：全尺寸 MLP，处理"困难"token。
  - 路由器：基于前一层隐状态预测 token 难度。
  - 意义：提供了研究"FFN 的计算量是否可以按 token 动态调整"的系统化框架。

### 理论支撑

- **Kolmogorov-Arnold 表示定理** (1957): 任何定义在 $[0,1]^n$ 上的多元连续函数 $f$ 都可以表示为 $f(x_1,...,x_n) = \sum_{q=0}^{2n} \Phi_q(\sum_{p=1}^{n} \phi_{q,p}(x_p))$。KAN 直接实现了这一表示。
- **MLP 的万能近似 vs KAN 的近似效率**: 虽然 MLP 也是万能近似器，但 KAN 在特定函数类上的近似效率（所需参数量）可能更优——特别是对具有可分离结构的函数。
- **自适应计算的帕累托最优**: Duo-LLM 探索的是计算-精度 Pareto 前沿——给定总计算预算，如何在不同 token 之间最优分配。

### 待解决问题

- **KAN 在大规模 LLM 中是否可行？** 目前 KAN 仅在小规模实验中验证。B-spline 的计算开销和内存需求在数十亿参数规模下是否可接受？是否存在更高效的一元函数参数化方案？
- **KAN 与 Attention 的兼容性**: KAN 作为 Channel Mixer 与标准 Attention 组合时，是否需要调整训练策略？
- **Duo-LLM 的路由准确性**: "token 难度"如何定义和度量？路由器的决策是否与人类直觉一致（如停用词 = 简单，专业术语 = 困难）？
- **替代方案的公平比较**: KAN vs MLP 的比较在参数量和 FLOP 两个维度上如何控制？等参数量 vs 等 FLOP 可能得出不同结论。

### 对未来的影响

- **KAN 更可能作为 MLP 的补充而非替代**: 在需要高精度函数逼近的特定子模块中使用 KAN，其余部分保持 MLP，形成混合架构。
- **"一种 FFN 服务所有 token"的范式可能终结**: Duo-LLM 和 MoE 都指向同一方向——不同 token 应该获得不同复杂度的 Channel Mixing。未来可能不存在统一的"FFN"概念，而是一个计算复杂度连续可调的特征变换模块。
- **可解释性驱动的架构设计**: KAN 的可解释性优势可能在科学计算、医疗等高风险领域找到独特应用。

### 潜在技术方向

- **高效 KAN 实现**: 用多项式近似或分段线性函数替代 B-spline，降低计算成本。FourierKAN、WaveletKAN 等变体。
- **KAN-MLP 混合层**: 在同一层中并行使用 KAN 和 MLP 路径，路由器决定每个 token 使用哪条路径。
- **连续计算强度 FFN**: 将 Duo-LLM 的二元选择（轻量/标准）推广为连续的计算强度调节——如动态宽度 MLP（根据 token 难度调整隐层维度）。
- **基于检索的 Channel Mixing**: 不用固定的参数矩阵进行特征变换，而是从外部知识库检索相关的变换向量，实现"无限容量"的 Channel Mixer。
