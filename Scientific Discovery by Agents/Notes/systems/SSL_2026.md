# SSL 2026

## 基本信息
- **论文**: From Skill Text to Skill Structure: The Scheduling-Structural-Logical Representation for Agent Skills
- **作者**: Qiliang Liang, Hansi Wang, Zhong Liang, Yang Liu (Peking University)
- **年份**: 2026
- **来源**: arXiv:2604.24026v4
- **关键词**: agent-skill, representation-learning, skill-discovery, risk-assessment, structured-knowledge, cognitive-linguistics

## 核心思想
大语言模型（LLM）智能体越来越依赖可复用技能——将指令、控制流、约束和工具调用组合在一起的能力包。然而，当前智能体系统中的技能仍以文本为主（主要是 SKILL.md 风格文档），机器可用的信号大多嵌入在自然语言描述中。这导致技能管理和使用时，调用接口、执行结构和具体副作用等信息被纠缠在单一文本表面。为此，作者借鉴 Schank 和 Abelson 在认知语言学中的经典理论（记忆组织包、脚本理论和概念依赖），提出了首个面向智能体技能的结构化表示——调度-结构-逻辑（SSL）表示。SSL 通过 LLM 归一化器将现有 SKILL.md 文件转化为三层结构。在技能发现和风险评估两项任务中，SSL 显著优于纯文本基线：技能发现中 MRR@50 从 0.649 提升至 0.729；风险评估中宏 F1 从 0.409 提升至 0.509。

## 研究问题与动机
a) 随着 LLM 智能体系统从单次工具调用走向多步骤、跨任务的复杂工作流，可复用技能（skill）成为核心抽象。但技能仍以 SKILL.md 等文本文件形式存在，机器需要从冗长自然语言中推断调用接口、执行阶段和资源使用等关键信号。作者的核心驱动力是：**为技能提供一个结构化、机器可操作的中间表示层**，使下游系统无需每次重新解析全文即可获取技能的关键证据。

b) 现有方法的痛点：
- 调用接口、执行结构、操作/资源使用证据被压缩在单一文本表面，语义上不同的属性被混在一起；
- 技能发现需要超越稀疏元数据的信号，纯文本检索难以区分功能相似的技能；
- 风险评估需要在执行前检查数据外泄、权限提升等风险，但文本形式使这些信号难以被系统化检查；
- 缺乏一个 **可复用的、来源锚定的表示层**，使下游任务能直接读取类型化证据而不需反复解析全文。

c) 研究假设：受认知语言学中记忆组织包（目标级上下文）、脚本理论（有序执行场景）和概念依赖（原始动作结构）的启发，如果将一个技能文档解耦为三个独立但关联的层次——调度层（何时调用）、结构层（如何组织工作）、逻辑层（执行什么操作和资源），那么这种显式结构化表示将使技能更易于被机器搜索、检查和复用。

## 方法细节
a) **Pipeline 流程：输入 → 处理 → 输出**

- **输入**：原始 SKILL.md 技能文档 d
- **处理**：LLM 归一化器（基于 DeepSeek-V3.2）执行四阶段归一化流水线：
  1. **Pass 1 (技能记录提取)**：从源文档提取调度层记录 r_sch，包括 skill_goal、intent_signature、expected_inputs/outputs、dependencies、tags、control_flow_features、entry_scene_id 和 subscenes
  2. **Pass 2 (场景分解)**：将技能分解为 2-5 个宏观执行场景（scene），每个场景标注类型（PREPARE/ACQUIRE/REASON/ACT/VERIFY/RECOVER/FINALIZE）、数据约定、进入/退出条件和场景间转换规则
  3. **Pass 3 (逻辑步骤展开)**：将每个场景展开为原子逻辑步骤（logic step），分配 act_type（READ/SELECT/COMPARE/VALIDATE/INFER/WRITE/UPDATE_STATE/CALL_TOOL/REQUEST/TRANSFER/NOTIFY/TERMINATE）、resource_scope（MEMORY/LOCAL_FS/CODEBASE/PROCESS/USER_DATA/CREDENTIALS/NETWORK/OTHER）、参数、效果和资源目标
  4. **Pass 4 (验证)**：硬验证（JSON 可解析、ID 唯一、枚举值有效、包含链接和转换目标合法）和软检查（场景输出是否有逻辑步骤绑定、数据流引用是否一致）
- **输出**：类型化 JSON 图 G_d = (r_sch, G_str, G_log, R_cont, R_entry)，包含三层和跨层包容关系/入口指针

b) **三层结构设计**：
- **调度层 (Scheduling)**：技能级接口记录，对应技能"是什么、何时用"。包含 goal、tags、intent_signature、expected_inputs/outputs、dependencies、top_pattern、control_flow_features 等字段。每个技能一个记录，可在仓库级别进行比较而无需展开完整图形。
- **结构层 (Structural)**：场景级有向图，节点是执行阶段（scene），边是阶段间转换。将底层操作分组为准备、获取、推理、执行、验证、恢复等连贯阶段，使技能的阶段组织在检查逻辑步骤之前即可见。
- **逻辑层 (Logical)**：逻辑步骤有向图，节点是原子动作，边是步骤间微级转换。每个步骤标注动作类型（12 种闭集）、资源范围（8 类闭集）、参数绑定、前提条件、效果和下一步规则。原子性按源文档支持的最小操作单元定义。

c) **关键设计原则**：
- **紧凑 (Compact)**：仅保留技能管理和使用所需证据，排除主观属性
- **类型化 (Typed)**：使用受限词汇表使输出可跨技能比较
- **锚定 (Grounded)**：所有字段严格总结源文档中的证据，不推断隐藏行为
- 归一化器使用 DeepSeek-V3.2，temperature=0.1，初始用 thinking 模式，失败后用 non-thinking 重试
- 最终归一化产率：6,300 个技能目录中 6,184 个产生有效 SSL 记录（98.16%）
- 人工审计 100 个技能，83% 的 SSL 字段被判定为受源文档支持

## 与其他方法对比
a) 本质不同：
- 与纯文本 SKILL.md 相比：SSL 将纠缠在自然语言中的三类信号（调用/结构/操作）显式化为类型化字段
- 与工具检索压缩/增强方法（EASYTOOL、CRAFT）相比：SSL 不改变检索模型，而是**在表示层面**提供结构化证据
- 与知识图谱方法（Graph of Skills）相比：SSL 聚焦于**单个技能内部**的结构化，而非技能间依赖关系
- 与 AGENTS.md 相比：SSL 面向技能本身而非代码仓库的智能体文档

b) 创新点：
- 首次为智能体技能设计结构化表示，借鉴认知语言学理论作为设计类比
- 将技能文档解耦为调用接口、执行阶段和原子操作三层，每层使用闭集词汇表
- LLM 归一化器实现从非结构化文本到类型化 JSON 的转换，支持 98%+ 的技能
- 表示与源文档互补而非替代，在技能发现和风险评估中均验证有效

c) 适用场景：
- **技能发现/检索**：结构化接口信号显著提升意图级匹配
- **风险评估**：显式化动作类型和资源范围帮助识别数据外泄、凭证访问等风险
- **技能选择/组合**：场景级结构支持技能间比对和组合
- **执行检查点**：逻辑层提供操作级跟踪能力

d) 方法对比总结：

| 维度 | 纯文本 SKILL.md | SSL 表示 | SSL + SKILL.md 联合 |
|------|-----------------|----------|---------------------|
| 调用接口可读性 | 需从全文推断 | 类型化字段显式给出 | 最佳：结构与上下文互补 |
| 执行结构可见性 | 隐藏在段落中 | 场景图显式展示 | 场景图 + 原始说明 |
| 资源/风险信号 | 分散在文本中 | act_type + resource_scope 类型化 | 结构化信号 + 源文档语义解读 |
| 技能发现 MRR@50 | 0.588-0.649 | 0.694-0.729 (SSL-Rich) | — |
| 风险评估宏 F1 | 0.235-0.409 | 0.422 (Full SSL alone) | 0.509 (MD + SSL) |
| 可维护性 | 人类可读，机器难处理 | 机器可操作，需归一化器维护 | 互补关系 |

## 关键结果
a) 实验设计：
- **技能发现**：6,184 个技能作为候选池，431 个意图级查询（功能型 95、约束型 86、组合型 88、安全型 73、场景型 89），使用 Qwen3-Embedding-0.6B + FAISS 内积索引，评估 MRR@50、NDCG@5/10、Recall@10
- **风险评估**：252 个分层抽样的技能，6 个风险维度（数据外泄、破坏性行为、权限提升、隐蔽执行、资源滥用、凭证访问），金标由 Gemini-3.1-pro + Claude-Sonnet-4.5 + GPT-5 三模型多数投票生成，评估使用 DeepSeek-V3.2 固定评判器

b) 关键数据：
- **技能发现**：
  - Desc + SSL-Rich 达到最高 MRR@50: 0.729，相比最强非 SSL 基线 Desc + Source Outline (0.649) 提升 0.080
  - 相比 Desc Only (0.588) 提升 0.141 (+24.0%)
  - 相比 Full SKILL.md (0.645) 提升 0.084 (+13.0%)，说明增益非来自简单嵌入更多文本
  - 消融实验：SSL-Shallow (0.716) > SSL-Sched (0.694)，SSL-Rich 最佳 (0.729)，说明场景级和接口级信号最关键
  - Bootstrap 95% CI: [0.051, 0.111]
- **风险评估**：
  - Full SKILL.md + SSL 宏 F1: 0.509，相比 Full SKILL.md alone (0.409) 提升 0.101 (+24.4%)
  - 最大增益维度：Credential Access (0.695→0.780)、Resource Abuse (0.271→0.419)、Data Exfiltration (0.511→0.699)
  - Full SSL alone 宏 F1 (0.422) 仅略高于 Full SKILL.md (0.409)，表明主要收益来自结构化证据作为源文档补充
  - Bootstrap 95% CI: [0.051, 0.152]

c) 最优势场景：
- **意图级技能搜索**：当查询涉及功能、约束、组合和领域特定术语时，SSL 的 tags、goal、scene types 和 resource scopes 提供精准匹配信号
- **操作性风险检测**：涉及数据流、凭证、网络访问和资源消耗的风险维度受益最大，因为 SSL 显式化这些操作和资源证据

## 局限性
- **静态性**：SSL 从静态文档提取，无法判断运行时动态行为（如代码生成命令、条件资源访问）
- **归一化器忠实度**：LLM 归一化可能遗漏事实、过度规范化，或对不明确技能映射到粗粒度枚举值（83% 支持准确率）
- **评估范围**：仅验证了技能发现和风险评估，未评估 SSL 对实际智能体规划、执行或监控的影响
- **风险评估金标**：由模型生成而非人类专家审计，结果应视为静态风险识别性能
- **SSL 不能替代源文档**：案例研究表明，当安全语义在原始文本中更清楚时，SSL 可能反而降低风险判断质量

## 学习与应用
a) **开源情况**：SSL 指南、标注技能语料库和评估数据集已在 GitHub 发布（github.com/COOLPKU/SSL），数据/代码包在论文接收时正式释放

b) **关键实现细节**：
- 归一化器后端：DeepSeek-V3.2，初始 pass 用 thinking 模式 + temperature 0.1，重试 pass 用 non-thinking + temperature 0.1
- 每个技能最多 5 次 API 尝试（初始 pass）+ 3 次重试
- 硬验证：JSON 可解析、所有必需字段存在、全局唯一 ID、有效枚举值、有效包容链接、有效入口指针、转换目标合法
- 提示约束：仅输出裸 JSON，不带 Markdown 代码块、解释或对话前缀
- Schema 闭集词汇表：7 种 scene_type、12 种 act_type、8 种 resource_scope

c) **可迁移性**：
- SSL 设计原则（紧凑、类型化、锚定）可推广到其他需要非结构化→结构化转换的智能体文档场景
- 归一化器流水线可作为通用的 NL2JSON 转换框架迁移
- 三层分解理念可应用于工具文档、API 规范等其他机器可操作表示场景

## 核心贡献
a) **一句话核心思想**：将技能文本解耦为调用/执行/操作三层类型化图结构。

b) **速记版 Pipeline**：
1. 输入原始 SKILL.md 文本
2. 用 LLM 提取技能的调用接口信息（目标、签名、标签、依赖等）
3. 将技能内容分解为有序的执行场景（准备→获取→推理→执行→验证→完成）
4. 将每个场景展开为原子逻辑步骤（标注动作类型、资源范围、参数绑定和效果）
5. 验证 JSON 结构和内部一致性，输出三层有向图

c) **与本综述的具体关联**：
- 属于智能体技能的**表示层**工作，是技能发现、技能管理和技能安全的基础设施
- 可放在综述的"Agent Skill Architecture"或"Skill Representation & Management"章节
- 支持论点：结构化的中间表示是使技能生态系统（管理、发现、组合、安全审查）可扩展的关键技术
- SSL 本身的归一化器也是 LLM 辅助知识提取的一个实例，连接"Agents for Science"和"Science for Agents"两条线

## 与综述的关联
SSL 在综述框架中处于**技能表示基础设施**的位置——它是连接原始技能文档与下游技能中心系统（检索、路由、安全审查、组合）的中间层。这意味着 SSL 是"Skills as First-Class Agent Artifacts"这一趋势中关键的"表示规范化"工作，与 SkillOS（技能管理）和 ARA（研究产物结构化）形成互补：SSL 定义技能的静态结构，SkillOS 定义技能的动态演化，ARA 定义研究产物的完整结构化协议。
